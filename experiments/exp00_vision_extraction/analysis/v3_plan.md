# Prompt v3 변경 계획 — GT 엑셀 양식과 1:1 일치

## 동기

v2 평가에서 발견한 두 종류의 비용:

1. **이름·라벨 노이즈** — `is_discounted`/`has_discount`, `brand_name`/`brand`, `"모델 착용샷"`/`"모델착용샷"`, `delivery_info`(방식)/`delivery_fee`(일정) 같이 **의미는 같은데 키가 달라서** evaluate에 정규화·매핑 코드를 끼워 넣음. 이런 코드는 디버깅 부담 + 다음 사람이 양식 어긋남을 또 발견해야 함.

2. **정의 오버랩** — 마케팅 트리거(과탐지), wishlist_count(팔로워 혼동) 같이 프롬프트가 광범위해서 모델이 보수적이지 못한 경우.

v3에서 둘 다 해결.

## 변경 사항

### A. 키 이름 GT 엑셀과 동일

| v2 키 | → v3 키 | GT 엑셀 컬럼 |
|-------|--------|--------------|
| `is_discounted` (boolean) | **`has_discount`** (0/1) | `has_discount(0/1)` |
| `brand_name` | **`brand`** | `brand` |
| `delivery_info` (방식) | **`delivery_info`** (일정) — 정의 변경 | `delivery_info` (사용자 GT는 일정) |
| (없음) | **`delivery_fee`** (배송 방식·비용) | `delivery_fee` |
| `style_keywords` (array) | (그대로 유지) | `style_keyword_1/2/3` (loader가 합침) |

### B. shot_type 라벨

v2: `"모델 착용샷", "흰 배경 단독샷", "행거샷", "기타"`
**v3**: `"모델착용샷", "단독샷", "행거샷", "기타"` (GT 엑셀의 실제 사용 라벨)

### C. wishlist_count 명확화

```
**wishlist_count**: 상품 카드의 ♡(하트) 옆에 표기된 찜/좋아요 수를 추출.
- ❌ 브랜드명/스토어명 옆에 표시되는 큰 숫자(예: "260.1만")는 브랜드 팔로워이므로 무시
- ❌ 상품 상세페이지가 아닌 광고/추천 영역의 숫자 무시
- 한글 단위 변환: "9.2만" → 92000, "1.4천" → 1400, "5.3K" → 5300
- 지그재그는 좋아요 기능이 없으므로 항상 null
```

### D. 마케팅 트리거 좁히기

#### trend_hype (인기/유행)
허용 키워드 한정 (외 모두 0):
```
"BEST", "랭킹", "1위", "인기", "품절임박", "재입고", "주문폭주", "HOT",
"대란", "핫딜", "트렌드", "리오더", "REORDER", "RANKING"
```
**제외(자주 잡히는 false positive)**: 카테고리 기술어 (`핏보장`, `모델소장`, `여친룩`, `데이트룩`, `코디추천`)

#### bundle (묶음/증정)
허용 키워드:
```
"1+1", "2+1", "세트", "증정", "사은품", "구성", "추가증정", "포함" (단 "벨트포함", "안감포함" 같은 디자인 설명은 제외)
```

#### confidence (신뢰/검증)
허용 키워드:
```
"누적판매", "재구매율", "MD추천", "리얼후기", "후기 N건", "베스트리뷰",
"인증", "특허", "공식"
```

#### marketing_phrases
유지. 위 3축에서 1로 판단된 실제 키워드만 list로.

### E. discounted_price 가이드 강조

v2 안내문에 이미 있음:
> 쿠폰 할인 ≠ 기본 할인 (쿠폰가는 무시)

v3에서 더 명시:
```
- 화면에 "쿠폰적용가" / "쿠폰가" / "회원가" 표시가 있으면 → 그 가격은 무시
- 메인 할인 표시(취소선 원가 + 할인가)만 사용
- 메인 할인 없고 쿠폰가만 있으면 → has_discount=0, discounted_price=null, discount_rate=null
```

## evaluate.py 변경

v3 결과부터는:
- `_norm_shot_type()` 정규화 함수 **불필요** (라벨 일치)
- `PRED_TO_GT_KEY` 매핑 **불필요**
- `delivery_info+delivery_fee` 합쳐 비교 로직 **불필요** (양쪽이 동일 의미)

다만 v2 결과에서도 동일 코드로 평가 가능하게 **하위 호환 가드** 유지 권장.

## 변경 미적용 — 추출 시간/비용 예산

v3 재추출은 다음 사이클에. v2 결과 99/99로 평가 완료된 상태이므로 **분석·v3 설계 → 의사결정 → 한 번에 v3 추출**이 효율.
